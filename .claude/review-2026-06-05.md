## Critical

### 1. No tests for security-critical event subscribers

**Files:** `LoginRedirectSubscriber.php`, `BeaconAccessControllerSubscriber.php`

Both subscribers are completely untested. A `grep` across `tests/` returns zero matches for either class name. These handle authentication redirect and authorization — regressions here are security regressions.

**Fix:** Add dedicated unit tests for both subscribers. For `LoginRedirectSubscriber`, cover: AccessDeniedException on admin route → redirect; AccessDeniedException on non-admin route → 403 passthrough; authenticated user with insufficient role → 403 (not redirect); login route itself → no redirect loop. For `BeaconAccessControllerSubscriber`, cover: class-level `#[BeaconAccess]`, method-level, both combined, and non-matching controllers.

### 2. `\Throwable` catch swallows fatal errors — `BeaconAccessControllerSubscriber:58`

```php
} catch (\Throwable) {
    $isGranted = false;
}
```

Catching `\Throwable` suppresses `\Error` (TypeError, OutOfMemoryError, etc.). A mismatched voter signature or missing service would silently become "access denied" instead of surfacing the real bug.

**Fix:** Catch `\Exception` or the specific exceptions `isGranted()` can throw (typically `AuthenticationException`, `AccessDeniedException`).

## High

### 3. `$request->getSession()` may throw — `LoginRedirectSubscriber:68`

```php
$this->saveTargetPath($request->getSession(), $this->firewallName, $request->getUri());
```

If the request has no session (stateless sub-request, or before session is started), `getSession()` throws. The subscriber runs on `KernelEvents::EXCEPTION` at priority 2, so an unhandled exception here would replace the original `AccessDeniedException` with a runtime error, breaking the error flow.

**Fix:** Wrap in `if ($request->hasSession())`, or catch the exception and let the original AccessDeniedException propagate.

### 4. `$firewallName` is hardcoded — `LoginRedirectSubscriber:31`

```php
private readonly string $firewallName = 'admin',
```

Unlike `loginRoute`, `routePrefix`, and `locales`, the firewall name is not configurable. If a consuming app uses a different firewall name (e.g., `'main'`, `'secured'`), `saveTargetPath` will use the wrong key and the `LoginFormAuthenticator` won't redirect back after login.

**Fix:** Add `$firewallName` to the bundle config tree and pass it from `BeaconAdminBundle::loadExtension()`.

## Medium

### 5. Regex path matching — `LoginRedirectSubscriber:51-52`

```php
if (!str_starts_with($path, $this->routePrefix)
    && !preg_match('#^/[a-z]{2}(_[A-Z]{2})?'.preg_quote($this->routePrefix, '#').'#', $path))
```

The locale regex `[a-z]{2}(_[A-Z]{2})?` is correct for ICU format. `preg_quote()` escapes the prefix properly. No ReDoS risk — the regex is anchored and the variable portion is quoted. This is **not** a vulnerability, but worth noting the approach is sound.

**Minor improvement:** Consider appending `/?` to the route prefix in the regex to handle trailing slashes consistently.

### 6. `strtok` on empty string — `LoginRedirectSubscriber:77`

```php
$locale = strtok($locale, '_');
```

If `$request->getPreferredLanguage()` returns an empty string (no Accept-Language header), `strtok('', '_')` returns `false` (PHP ≤8.3) or `''` (PHP 8.4+). On PHP <8.4, `$locale` would be `false`, which casts to empty string in the URL generator — functionally harmless but messy.

**Fix:** Add a guard: `$locale = $locale ? strtok($locale, '_') : 'en';`

### 7. Repeated `(bool)` casts — `BeaconAdminBundle:159,166`

```php
if ((bool) $config['security']['voters']) { ... }
if ((bool) $config['security']['redirect_to_login']) { ... }
```

Both nodes are `booleanNode` in the config tree — the cast is redundant. Not a bug, just noise.


Using `\` prefix for global functions is a micro-optimization (avoids namespace lookup). This is fine but inconsistent with other files like `LoginRedirectSubscriber` which doesn't use them. Pick one style.

Repeated across 14+ test methods. Consider extracting to a `createAdminUser(): InMemoryUser` helper to reduce duplication and centralize the test password.
